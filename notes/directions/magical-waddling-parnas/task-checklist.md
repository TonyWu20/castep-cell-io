### lib-tdd assessment: Test code specification quality

**The `test_code` has TWO problems that would prevent compilation:**

1. **BUG: `KpointsMpSpacing(0.05)` is not valid Rust.** `KpointsMpSpacing` is defined as a struct with **named fields** (`pub value: f64, pub unit: Option<InvLengthUnit>`), not a tuple struct. The test `build_rejects_all_three_kpoint_specs` at line 210 of the test code uses `KpointsMpSpacing(0.05)` which is invalid syntax. The correct construction would be `KpointsMpSpacing { value: 0.05, unit: None }`.

2. **BUG: Multiple import paths reference types through private submodules that are not publicly re-exported.** In Rust, items inside `mod foo;` (private) are not accessible from sibling/higher modules unless re-exported with `pub use`. The test code has five imports that will fail to compile:

| Import path in test code | Private module | Re-exported? |
|---|---|---|
| `crate::cell::positions::positions_frac::PositionFracEntry` | `positions_frac` (private) | NO — only `PositionsFrac` is re-exported from `positions/mod.rs` |
| `crate::cell::bz_sampling_kpoints::kpoint::Kpoint` | `kpoint` (private) | YES — but import path bypasses re-export; should use `crate::cell::bz_sampling_kpoints::Kpoint` |
| `crate::cell::bz_sampling_kpoints::bs_kpoint_path::BsKpointPathEntry` | `bs_kpoint_path` (private) | NO — `BsKpointPathEntry` is not re-exported anywhere |
| `crate::cell::symmetry::symmetry_ops::SymmetryOp` | `symmetry_ops` (private) | YES — but import path bypasses re-export; should use `crate::cell::symmetry::SymmetryOp` |
| `crate::cell::phonon::phonon_kpoint_path::PhononKpointPathEntry` | `phonon_kpoint_path` (private) | NO |
| `crate::cell::phonon::phonon_kpoint_list::PhononKpointListEntry` | `phonon_kpoint_list` (private) | NO |

   Three of these (`PositionFracEntry`, `BsKpointPathEntry`, `PhononKpointPathEntry`, `PhononKpointListEntry`) are not re-exported at all. Two of them (`Kpoint`, `SymmetryOp`) are re-exported but the import uses the private module path instead of the re-export path. The test code must either:
   - Use public re-export paths (e.g., `crate::cell::bz_sampling_kpoints::Kpoint`)
   - Or the parent modules must be modified to add missing `pub use` re-exports (which would be outside `cell_document.rs`)

3. **`PhononKpointPathEntry` is constructed directly as a struct literal** — this is fine because `PhononKpointPathEntry` is `pub` and has public fields (`coord: [f64; 3]`). No `bon::Builder` needed. The issue is purely the import path.

4. **`SpectralKpointPathEntry` import works** — it IS re-exported from `bz_sampling_kpoints/mod.rs` as `pub use spectral_kpoint_path::SpectralKpointPathEntry;`. So `crate::cell::bz_sampling_kpoints::spectral_kpoint_path::SpectralKpointPathEntry` would still fail (private module path), but `crate::cell::bz_sampling_kpoints::SpectralKpointPathEntry` would work. However, the test_code uses the private module path. Fixing the import to use the re-export path resolves this.

Despite the import bugs, the **structure of the tests is a valid specification**: 10 tests with concrete assertions (`is_err()`, `is_ok()`) that exercise each mutual-exclusion rule, the offset exception, and the happy path. The tests are falsifiable and would drive the correct implementation once the import/constructor bugs are fixed.

**Signatures match**: The `signature` field describes the custom `build()` method signature, and the test code calls `CellDocument::builder()...build()` which exercises that method. Consistent.

## Overall Assessment

**NEEDS MORE DETAIL AND BUG FIXES BEFORE IMPLEMENTATION**

### What must be fixed before TASK-1 can be implemented:

1. **Fix `KpointsMpSpacing(0.05)` construction** in the `test_code` for `build_rejects_all_three_kpoint_specs`. Change to `KpointsMpSpacing { value: 0.05, unit: None }`.

2. **Fix all private-module import paths** in the `test_code`:
   - `crate::cell::bz_sampling_kpoints::kpoint::Kpoint` — change to `crate::cell::bz_sampling_kpoints::Kpoint`
   - `crate::cell::symmetry::symmetry_ops::SymmetryOp` — change to `crate::cell::symmetry::SymmetryOp`
   - `crate::cell::positions::positions_frac::PositionFracEntry` — either add a `pub use` re-export in `positions/mod.rs`, or import differently. Since this task says changes are confined to `cell_document.rs`, the former requires scope expansion. Alternatively, use `PositionFracEntry::builder()` instead of direct struct construction to avoid needing the type in scope directly (the builder is accessible without importing the type if it's used method-style on a constructed builder). But actually the test constructs a `PositionsFrac` struct directly which needs `PositionFracEntry` in scope.
   - `crate::cell::bz_sampling_kpoints::bs_kpoint_path::BsKpointPathEntry` — needs a `pub use` re-export in `bz_sampling_kpoints/mod.rs` (outside current scope), OR use `BsKpointPathEntry::builder()` instead of direct construction, which still needs the type in scope.
   - `crate::cell::phonon::phonon_kpoint_path::PhononKpointPathEntry` — needs either a re-export or an alternative approach.
   - `crate::cell::phonon::phonon_kpoint_list::PhononKpointListEntry` — same issue.

   The cleanest fix is either: (a) add missing `pub use` re-exports in the respective `mod.rs` files, or (b) change the test code to use the public re-export paths where they exist, and use builder-pattern construction (via `::builder()` which is available even if the type is not directly imported) for the three types that lack re-exports. Option (b) requires verifying that the builder pattern is callable without the type being in scope (it isn't — you need to import the type to call `Type::builder()`).

3. **Fix or remove the erroneous "README line 35-36" wiring checklist item** in TASK-1. The README is empty; there is nothing to verify.
