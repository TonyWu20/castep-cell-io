## 1. Key Architectural Patterns from the Plan

### Bon Fallible Builders (Custom Finishing Function)

The central pattern in this plan is converting `CellDocument`'s builder from an infallible `build()` to a fallible one using bon's "Custom Finishing Function" approach:

- The `#[derive(Builder)]` is modified with `finish_fn(vis = "", name = build_internal)` to rename the generated method.
- A manual `impl` block on `CellDocumentBuilder<S: IsComplete>` provides a public `build(self) -> CResult<CellDocument>` that calls `self.build_internal()` then runs validation checks.
- The `IsComplete` trait lives in the private `cell_document_builder` module generated by the derive. It resolves from the parent module because both the derive and the manual impl are in the same parent module (`cell_document`).

### Structurally Enforced Mutual Exclusions

Some mutual-exclusion rules are already enforced at the type level via enums (e.g., `Lattice` enum for `LATTICE_CART`/`LATTICE_ABC`). The plan adds runtime validation for the remaining five rules plus one absorbed deferred item (D2).

### Parse-Time Guards

Beyond builder validation, the plan adds parse-time guards in `from_cell_file` to prevent spectral/BS_ duplication at the source: BS_ blocks are only parsed when the corresponding spectral field is `None`.

## 2. Known Failure Modes to Avoid

### IsComplete Trait Path Resolution

The manual `build()` impl must reference `cell_document_builder::IsComplete`. This is a private module generated by the derive macro. In bon v3.9.1, this resolves correctly when the impl block is in the same parent module as the derive. **If bon's code generation changes** (module naming convention), this will break silently. Pin the bon version or add a compile-test.

### BS_ Block Duplication

The original code unconditionally populated both `spectral_kpoint_path` and `bs_kpoint_path` when a `.cell` file used BS_ block names. The fix adds guards (`if spectral_kpoint_path.is_some() { None } else { ... }`). **Failure mode**: if a new BS_-like alias is added later and the guard is forgotten, the builder's mutual-exclusion check will catch it at runtime. The order of parse operations matters — spectral blocks must be parsed before BS_ blocks.

### Offset Fields Are Not Mutually Exclusive

`kpoints_mp_offset` and `spectral_kpoints_mp_offset` are tuning parameters, not mutually exclusive with their grid/list counterparts. They must NOT be included in the count checks. The verification tests include a dedicated test (`build_allows_mp_offset_with_grid`) to guard against accidental inclusion.

## 3. Deferred Improvements That Should Be Incorporated

The following items from the Phase 1 deferred list (`notes/pr-reviews/phase-1/deferred.md`) were **not** addressed by this plan. They remain valid candidates for future phases:

| ID | Item | Notes |
|----|------|-------|
| D1 | Group `CellDocument` fields into sub-structs (KpointsParams, SpectralParams, PhononParams) | ~50 fields is unwieldy. Sub-structs would reduce surface area and isolate parsing/serialization. |
| D3 | Integration test with full round-trip fixture | Parse a .cell file with all keyword types, serialize back, parse again, compare. The existing `test_parse_mg2sio4_cell` (`#[ignore]`) is a template. |
| D4 | Fix doc comments on `SpectralKpointsMpGrid` | Doc comment uses plural `SPECTRAL_KPOINTS_MP_GRID` but canonical `KEY_NAME` is singular `SPECTRAL_KPOINT_MP_GRID`. |
| D6 | `#[non_exhaustive]` on `Lattice` enum | Pre-1.0 this is acceptable, but should be added before stabilization for API safety. |
| D7 | Backfill G5 test coverage (~50 tests) | All 8 phonon MP/fine types have exactly 1 test each; their reference counterparts have 6-8 tests. Low risk, high resilience value. |
| D8 | `SymmetryTol` has zero tests | Pre-existing gap — no `#[cfg(test)]` module at all. |
| D9 | `PhononKpointPathEntry` builder parity | Has no `bon::Builder` derive while `SpectralKpointPathEntry` does. |
| D10 | `SpectralKpointsList PartialOrd` audit | Stray `PartialOrd` derive on `SpectralKpointsList` not present on other list types. |

### D2 Was Absorbed

D2 (mutual-exclusion validation between spectral and BS_ types) was absorbed into this plan's scope. The validation is handled by merging `bs_kpoint_path` and `bs_kpoints_list` into the ^4 count check, plus parse-time guards in `from_cell_file`.

### D5 Was Completed (Not Deferred)

D5 (fix parsing order — spectral before BS_) was identified as an immediate defect, not a deferred item. It was addressed as part of this implementation.
