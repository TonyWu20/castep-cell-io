## 1. Goal Decomposition and Design Decisions

### G1: Convert `CellDocument` builder to fallible (bon `finish_fn`)

**Scope**: `castep_cell_io/src/cell_document.rs`, `#[derive(Builder)]` attribute (line 238–239).

**Design decision** — Custom Finishing Function vs. separate `.validate()`:
- The project already has `ParamDocument` using a separate `.validate(self) -> Result<Self, String>` called after `.build()`.
- For `CellDocument`, the plan and user chose bon's Custom Finishing Function because it bakes validation into the builder's `build()` method directly — the caller cannot accidentally forget to call `.validate()`.
- The generated `build()` is renamed to `build_internal()` (private, infallible), and a manual `impl` block provides a public `build() -> CResult<CellDocument>` that calls `build_internal()` then runs validation.

**Type change on the derive**:
```rust
// Before:
#[derive(Debug, Clone, Builder)]
#[builder(on(Lattice, into), on(Positions, into))]

// After:
#[derive(Debug, Clone, Builder)]
#[builder(on(Lattice, into), on(Positions, into), finish_fn(vis = "", name = build_internal))]
```

Breaking effect: call sites that use `.build()` will now get `CResult<CellDocument>` instead of `CellDocument`. The README already shows `.build().unwrap()`, which remains correct. No downstream change in this repo.

**Implementation notes**:
- `vis = ""` makes `build_internal()` private — only visible within the parent module (`cell_document`).
- `name = build_internal` renames the generated method.
- The `IsComplete` trait lives in the generated `cell_document_builder` private module. It must resolve from the manual `impl` block in the same parent module.

### G2: Implement custom `build()` on `CellDocumentBuilder`

**Scope**: New `impl` block immediately after `CellDocument` struct definition (after line 441).

**Type signature**:
```rust
use cell_document_builder::IsComplete;

impl<S: cell_document_builder::IsComplete> CellDocumentBuilder<S> {
    pub fn build(self) -> CResult<CellDocument> { ... }
}
```

**Design decision — `use cell_document_builder::IsComplete`**:
The `use` import at the top of this impl block is necessary because the `cell_document_builder` module is private (generated by the derive macro). The path resolves relative to the parent module (`cell_document`), not the crate root. Placing the import inside the `impl` block body or at the top of the file both work — the plan uses it at the top of the impl block for locality.

**Validation rules and their implementation**:
| Rule | Fields checked | Error message keyword | Count target |
|------|---------------|----------------------|--------------|
| ^3 | `kpoints_list`, `kpoints_mp_grid`, `kpoints_mp_spacing` | `KPOINTS_LIST / KPOINTS_MP_GRID / KPOINTS_MP_SPACING` | ≤ 1 |
| ^4+D2 | `spectral_kpoint_path`, `spectral_kpoints_mp_grid`, `spectral_kpoints_mp_spacing`, `spectral_kpoints_list`, `bs_kpoint_path`, `bs_kpoints_list` | `SPECTRAL_* / BS_*` (all 6) | ≤ 1 |
| ^5 | `phonon_kpoint_path`, `phonon_kpoint_list` | `PHONON_KPOINT_PATH / PHONON_KPOINT_LIST` | ≤ 1 |
| ^6 | `symmetry_generate`, `symmetry_ops` | `SYMMETRY_GENERATE / SYMMETRY_OPS` | ≤ 1 |

**Fields explicitly excluded from mutual-exclusion checks**:
- `kpoints_mp_offset` — tuning parameter, valid alongside any ^3 field
- `spectral_kpoints_mp_offset` — tuning parameter, valid alongside any ^4 field
- `bs_kpoint_path_spacing` — tuning parameter, not mutually exclusive
- `spectral_kpoint_path_spacing` — tuning parameter, not mutually exclusive

**Error construction**:
```rust
Err(Error::Message(
    "Only one of KPOINTS_LIST, KPOINTS_MP_GRID, and KPOINTS_MP_SPACING may be specified".into()
))
```

Uses `Error::Message(String)` variant from `castep_cell_fmt::Error`. The string is `into()`-ed because `Error::Message` takes `String`.

**Counting idiom**:
```rust
let count = [field1.is_some(), field2.is_some(), ...]
    .iter()
    .filter(|&&x| x)
    .count();
if count > 1 { return Err(...); }
```

This is the established pattern in this codebase (used in `ParamDocument` validation). It is concise, self-documenting, and avoids mutable state.

### G3: Update `from_cell_file` — two changes

**Scope**: `castep_cell_io/src/cell_document.rs`, `impl FromCellFile for CellDocument` (lines 443–766).

#### G3a: Keep the lattice mutual-exclusion check (lines 486–491)

**Decision**: Keep the parse-time check for "Both LATTICE_CART and LATTICE_ABC are specified." This produces a specific, actionable error message at parse time rather than relying on the `Lattice` enum's structural enforcement. The existing code is left unchanged.

#### G3b: Add parse-time guards for BS_ blocks (lines 545–553)

**Decision**: `bs_kpoint_path` and `bs_kpoints_list` are backward-compat aliases for `spectral_kpoint_path` and `spectral_kpoints_list`. When a `.cell` file uses BS_ block names, the original code unconditionally populates both fields with the same data — creating an impossible-to-satisfy conflict at builder validation time.

**Fix**: Guard BS_ parsing behind `spectral_*.is_none()` checks:

```rust
let bs_kpoint_path = if spectral_kpoint_path.is_some() {
    None
} else {
    find_block_any(cells, &["BS_KPOINT_PATH", "BS_KPOINTS_PATH"])
        .ok()
        .map(|rows| BsKpointPath::from_block_rows(rows))
        .transpose()?
};

let bs_kpoints_list = if spectral_kpoints_list.is_some() {
    None
} else {
    find_block_any(cells, &["BS_KPOINT_LIST", "BS_KPOINTS_LIST"])
        .ok()
        .map(|rows| BSKpointList::from_block_rows(rows))
        .transpose()?
};
```

**Critical ordering constraint**: Spectral blocks are parsed at lines 529–543 (before BS_ blocks at lines 545–553). This ordering must be maintained — the `is_some()` guards depend on it. The original code already has this order (from a prior fix), so no reordering is needed here.

**Why this is needed despite builder validation**: Without these guards, `from_cell_file` would create a `CellDocument` where both `spectral_kpoint_path` and `bs_kpoint_path` are `Some(...)` with identical data when the input uses BS_ block names — builder validation would reject this, making every BS_ `.cell` file unparseable.

#### G3c: Replace struct literal with builder (lines 714–765)

**Decision**: Replace the field-by-field struct literal:
```rust
Ok(CellDocument {
    lattice,
    positions,
    kpoints_list,
    bs_kpoint_path,
    // ... 46 fields ...
})
```
with a builder call:
```rust
CellDocument::builder()
    .lattice(lattice)
    .positions(positions)
    .maybe_kpoints_list(kpoints_list)
    .maybe_bs_kpoint_path(bs_kpoint_path)
    // ... all 46 optional fields use maybe_* prefix ...
    .build()
```

**bon naming convention for optional fields**: bon generates setter names with a `maybe_` prefix for `Option<T>` fields. Required fields (`lattice: Lattice`, `positions: Positions`) use plain setters. The `on(Lattice, into)` and `on(Positions, into)` annotations mean the setter accepts `impl Into<Lattice>` and `impl Into<Positions>`, respectively.

## 3. Pattern Requirements

### P1: bon Custom Finishing Function

**Source**: bon documentation (Custom Finishing Function section).

**Pattern**: 
1. `finish_fn(vis = "", name = build_internal)` on the derive — renames the generated builder completion method.
2. Manual `impl<S: IsComplete> Builder<S> { pub fn build(self) -> Result<T, E> }` — provides the public API.
3. Inside `build()`: call `self.build_internal()`, validate the resulting struct, return `Ok` or `Err`.

**Precedent in this codebase**: None yet — this is the first use of bon fallible builders in the project. `ParamDocument` uses a separate `.validate()` method, which is the alternative pattern being deliberately NOT followed here.

### P2: IsComplete trait path resolution

**Pattern**: The `cell_document_builder::IsComplete` path
- `cell_document_builder` — a private module generated by `#[derive(Builder)]` on `CellDocument`
- `IsComplete` — a trait inside that module, implemented on `CellDocumentBuilder<S>` state types
- Resolution: the module is a child of the parent module (`cell_document`). The `use` statement at the top of the manual `impl` block resolves because both the derive output and the manual impl are in the same parent module.

**Failure mode**: If bon's module naming convention changes in a future version (e.g., from `cell_document_builder` to `__cell_document_builder`), the `use` path breaks. The `bon = "3.9.1"` version pin in `Cargo.toml` mitigates this. If upgrading bon later, run `cargo check` immediately.

### P3: Parse-time guard for BS_ aliases

**Pattern**: `let field = if sibling_field.is_some() { None } else { parse_block(...) };`
- Used to prevent spectral/BS_ field duplication.
- The sibling field (`spectral_*`) MUST be parsed before the guarded field (`bs_*`).

**Precedent in this codebase**: The `from_cell_file` method already uses a similar pattern implicitly — block names are searched with priority ("SPECTRAL_*" before "BS_*") in `find_block_any`. The guard makes this priority explicit and prevents double-population.

### P4: `.maybe_*` setter naming for optional fields

**Pattern**: bon automatically names setter methods for `Option<T>` fields with a `maybe_` prefix. This is a bon convention, not project-specific. For example, `kpoints_list: Option<KpointsList>` generates `.maybe_kpoints_list(value)`.

**Precedent in this codebase**: Used throughout the project where structs with `Option<T>` fields are built with bon (e.g., `ParamDocument` sub-group builders).

### P5: Test fixture construction with builders

**Pattern**: In tests, construct `CellDocument` instances via `CellDocument::builder()` with minimal valid required fields:
```rust
let result = CellDocument::builder()
    .lattice(minimal_lattice())
    .positions(minimal_positions())
    .maybe_kpoints_list(some_kpoints_list())
    .maybe_kpoints_mp_grid(some_mp_grid())
    .build();
```

**Precedent**: The `ParamDocument` test suite uses builder construction with `Default::default()` for sub-groups.

## 5. Known Pitfalls and Constraints

### Pitfall 1: bon `IsComplete` trait visibility
**Problem**: `cell_document_builder::IsComplete` is a private module generated by the derive macro. If the manual `impl` block is placed in a different module, the path won't resolve.
**Mitigation**: Keep the manual `impl` in the same parent module (`cell_document`) as the derive. The plan's placement (after the struct definition, same file) satisfies this.

### Pitfall 2: `.maybe_*` vs plain setter confusion
**Problem**: It is easy to accidentally use `.kpoints_list(value)` instead of `.maybe_kpoints_list(value)` for optional fields. The compiler will reject this — bon generates both setters only for non-optional fields after `on(_, into)` transformations.
**Mitigation**: If `cargo build` succeeds, the correct setters are used. The `from_cell_file` rewrite is large (~46 fields) but each setter error will produce a clear compile error.

### Pitfall 3: Missing optional field in from_cell_file builder
**Problem**: The struct literal has 46 optional fields. If one is forgotten in the builder call, bon's state machine will not reach `IsComplete`. However, since optional fields default to `None` in bon, forgetting a `maybe_*` setter simply leaves that field as `None` — no compile error, but the field silently becomes `None` even if parsed.
**Mitigation**: Cross-reference the builder call against the struct literal field-by-field. Copy the field list from this elaboration and check each one off.

### Pitfall 4: `bs_kpoint_path_spacing` is a tuning parameter
**Problem**: `bs_kpoint_path_spacing` sounds like it belongs with the BS_ k-point group, but it is NOT mutually exclusive with anything. It was already correctly excluded from the ^4 check in the plan. If someone later adds it to the check, it would cause false rejections.
**Mitigation**: The test `build_allows_mp_offset_with_grid` establishes the pattern. Consider adding a similar test for `bs_kpoint_path_spacing` if not already covered by existing tests.

### Pitfall 5: Test fixture construction requires valid LatticeCart/PositionsFrac
**Problem**: Tests need concrete `Lattice` and `Positions` values to build a `CellDocument`. These types have their own builders that require valid data. A minimal lattice vector (e.g., `[[10.0, 0.0, 0.0], [0.0, 10.0, 0.0], [0.0, 0.0, 10.0]]`) and positions (e.g., `[("Si", [0.0, 0.0, 0.0])]`) are needed.
**Mitigation**: Check `LatticeCart` and `PositionsFrac` constructors/builders before implementing tests. If they have complex builders, create helper functions in the test module.

### Pitfall 6: bon v3.9.1 specific behavior
**Problem**: The `finish_fn` and `IsComplete` trait behavior is bon v3.9.1-specific. Upgrading bon could change module naming or trait path.
**Mitigation**: `bon = "3.9.1"` is pinned in `Cargo.toml`. When upgrading bon, run `cargo check` and `cargo test` immediately to catch breaking changes.

## 6. Task Grouping Rationale

The plan decomposes into 3 task groups:

| Group | Name | Steps | TDD class | Dependencies |
|-------|------|-------|-----------|--------------|
| T1 | Builder fallible conversion | Steps 1-2 | lib-tdd | None (foundational) |
| T2 | from_cell_file updates | Step 3 | direct (plumbing) | T1 |
| T3 | Tests | Verification | mixed | T1+T2 for integration test |

**Rationale for grouping**:

- **T1 (Builder + validate)** is the foundational change. Everything depends on the builder returning `CResult<CellDocument>` instead of `CellDocument`. This is a natural, tight unit — the derive change and the custom `build()` impl are a single atomic change. Classified as **lib-tdd** because the validation logic is pure domain logic.

- **T2 (from_cell_file)** depends on T1 because it uses `.build()` which now returns `CResult`. The two sub-changes (BS_ guards and builder conversion) are tightly coupled within the same function body. Classified as **direct** because it wires parse infrastructure to domain validation — no pure logic to extract.

- **T3 (Tests)** has two sub-parts:
  - Unit tests (lib-tdd): can be written before implementation as they test the builder validation logic.
  - Integration test (plumbing): needs T1+T2 complete to exercise the parse→build pipeline.

**Recommended execution order**: T1 unit tests → T1 implementation → T2 implementation → T3 integration test. Run `cargo test -p castep_cell_io` after each group.
